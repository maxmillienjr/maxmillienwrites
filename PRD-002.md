# PRD-002 — Post-Scaffold Refinements

**Status:** Addendum to `PRD.md` (v1). Captures decisions made after the Phase 1 scaffold ship.
**Supersedes:** specific sections of v1 noted inline. v1 remains the historical record.

---

## Context

The Phase 1 scaffold is in place. Routes render, build is green. Reviewing the homepage spec against the scaffold surfaced four issues with v1 that need to be resolved before Phase 3 (homepage sections) starts:

1. The "Selected Work" rail conflated client case studies with open-source showcase repos — two different proof types.
2. Two of the three showcase cards were placeholders.
3. The `/resume` route's intent was ambiguous (PDF vs. HTML).
4. Boston University was incorrectly listed in the trust row (Max attended as a student; never employed there).
5. The nav had both an "Engage" page link and a "Book a Call" CTA — redundant.

---

## Change 1 — Split "Selected Work" into two rails

**Supersedes:** PRD.md §"Section 3 — Selected Work (the Case Study Rail)"

**Rationale:** Client case studies and open-source repos prove different things. A CTO reading the page should immediately understand "this is shipped client work" vs. "this is auditable code I can read right now." Lumping them obscures both signals.

### New Section 3 — Selected Work (Client)

Two cards. No GitHub links (codebases are private). Same drag-scroll gallery treatment as v1.

**Card 1 — Quiet Horizons TelePsychiatry**
> HIPAA-compliant Clinical OS for a concierge psychiatric practice. Ambient AI scribe transforms raw WebSocket audio into signed SOAP notes with zero draft-state writes — immutable audit trail by design. Angular 21, NestJS, Drizzle, Vertex AI Gemini, GCP BAA.

**Card 2 — PureTome AI Biographer**
> Consumer platform that helps authors write memoirs with an adaptive AI partner. Three-brain memory architecture (RAG + episodic + author profile) achieved a 40% narrative-coherence lift over baseline RAG. React 19, NestJS microservices, LangGraph, Neo4j, pgvector.

Each card links to `/work/[slug]` for the long-form case study.

### New Section 3b — Open Source (auditable proof)

Three cards. Each has a primary `View on GitHub →` link (opens in new tab) and a secondary `Read the architecture →` link to a `/work/oss/[slug]` deep-dive. Visual treatment slightly cooler than the client rail (monochrome, terminal-styled card chrome) so the audience reads them as "code artifacts" not "marketing case studies."

**Card 3 — `agent-native-monorepo`**
> A reference monorepo for building AI-native products without inheriting a stack you'll regret in six months. Yarn Berry workspaces, NestJS API, React 19 client, and a TypeScript AI engine package wired with LangGraph and Vertex AI. Demonstrates the Stateless Specialist Pattern: agents are pure functions over typed state, the orchestration graph is the only stateful surface, and every node is independently testable in isolation.
> **Stack:** Yarn Berry · TypeScript · NestJS · React 19 · LangGraph · Vertex AI · GitHub Actions
> *Extracted from PureTome's production monorepo. IP scrubbed per the Sanitization Rule — the skeleton, conventions, and orchestration patterns are real; the domain logic is generic.*

**Card 4 — `secure-data-vault`**
> A drop-in field-level encryption layer for Postgres + Drizzle ORM, designed for PHI/PII workloads under HIPAA or SOC 2. Transparent AES-256-GCM at the column level using Google Tink for key management, with envelope encryption against GCP KMS. Includes a deterministic-search variant for encrypted-column lookups without hauling the entire table into memory. The README includes the threat model, the audit-trail expectations, and what this *doesn't* protect against.
> **Stack:** TypeScript · NestJS · Drizzle · Postgres · Google Tink · GCP KMS
> *Extracted from Quiet Horizons' encryption-at-rest layer.*

**Card 5 — `realtime-voice-infra`**
> A production-grade pipeline for streaming audio dictation into structured AI output without buffering to disk. Browser captures audio → secure WebSocket → Google Speech-to-Text → Gemini for structured extraction → typed result back to the client. Includes back-pressure handling, reconnect semantics, and a stateless inference path that never persists draft transcripts (designed for regulated environments where draft-state writes are an audit liability).
> **Stack:** TypeScript · NestJS · WebSockets · Google Speech-to-Text · Vertex AI Gemini · Cloud Run
> *Extracted from Quiet Horizons' ambient scribe pipeline.*

## Change 2 — `/resume` page: HTML + PDF, both first-class

**Supersedes:** PRD.md §"Information Architecture" line for `/resume`

**Rationale:** Recruiters reflexively want a PDF. CTOs prefer in-browser skim. Building both is cheap and serves both audiences.

### Spec

- `/resume` renders an HTML version of the résumé using the same design system as the rest of the site (so it looks like a native page, not a PDF embed).
- A prominent `Download PDF ↓` button at the top right of the résumé header anchors a static `Millien-Max-2026.pdf` from `/public/resume/`.
- The HTML version is generated from a single source-of-truth: `src/content/resume.ts` (typed object). Both the page render and a future PDF-regeneration script consume the same object.
- For now, the PDF is the existing `/home/max/Desktop/Millien-Max-2026.pdf` copied into `public/resume/` — no automated PDF generation in v1.
- HTML résumé must be Lighthouse 100 SEO and printable (`@media print` styles that render cleanly if a user prints the HTML version directly).

### schema.org

Add `JsonResume`-style structured data on `/resume` so Google can parse the work history. Helps with rich results for `max millien resume`.

## Change 4 — Engage vs. "Book a Call" disambiguation

**Supersedes:** PRD.md §"Information Architecture" nav definition

### Old (redundant)

> **Nav (desktop, left-to-right):** Work · Stack · Engage · Resume · [Book a Call CTA]

### New (page vs. action)

> **Nav (desktop, left-to-right):** Work · Stack · Engage · Resume · Contact

**"Book a 30-min architecture call →" is a CTA, not a nav item.** It appears:
- In the hero (primary CTA)
- At the bottom of `/engage` (the conversion target — anchored as `/engage#book`)
- At the bottom of `/work` and `/work/[slug]` case studies (secondary CTA)

The `/engage` page is the *explanation* of how working with PureTome Labs works (rate, terms, week-1 deliverables, contract type). The CTA is the *action* of starting that conversation. Conflating them in the nav was the original mistake.

### `/engage#book` implementation

A Calendly inline embed (or equivalent — Cal.com is a fine free alternative). Anchor scroll behavior smooth-scrolls to the embed when the URL hash is present on load. Embed lazy-loads via `IntersectionObserver` to avoid the 200KB+ Calendly bundle hitting LCP.

## Change 5 — Showcase repo strategy reminder (not a v1 change, just making it explicit)

The three OSS repos do not need to exist *yet* for the cards to be designed. But they need to exist before the site goes live, otherwise the GitHub links 404 — which is worse than not having the cards at all. Implementation order:

1. Build the cards and route deep-dives in the codebase (`/work/oss/[slug]` MDX pages).
2. Before DNS cutover, ensure all three repos exist publicly under `github.com/maxmillienjr` with at minimum a README, license, and a "this is a work in progress, full release date X" note.
3. Phase 5 of the v1 milestone table is the natural place to do this.

This protects against the failure mode where the site ships, a CTO clicks `agent-native-monorepo`, and gets a 404.
