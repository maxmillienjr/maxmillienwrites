## Item 1 — Diagnose the hero mesh fallback

**Symptom:** user sees a 3×3 static grid of tech labels instead of the interactive WebGL knowledge graph.

**Not a code bug.** `src/components/hero/HeroMesh.tsx:244-245` correctly gates the canvas behind a `reduced || !hasWebGL()` check. The static grid is the designed-for fallback when either condition fires.

**Diagnostic procedure:**

1. Open the homepage in a desktop Chrome/Firefox/Safari tab.
2. Open DevTools → Elements.
3. Search the DOM for a `<canvas>` element inside the hero region.
4. Branch:
   - **Canvas absent** → user's OS has reduce-motion enabled. On GNOME: Settings → Accessibility → Reduce animation (toggle off). On macOS: System Settings → Accessibility → Display → Reduce motion. On Windows: Settings → Accessibility → Visual effects → Animation effects. This is **working as designed**; document and close.
   - **Canvas present but empty/blank** → real render bug. Open Console, capture any r3f/three.js warnings, and open a separate debugging session.
   - **Canvas present and rendering but looks wrong** → content bug in `HeroMesh.tsx` (node positions, materials, or animation loop). Separate debugging session.

**Resolution action:** no code change expected. If diagnosis reveals a real bug, spawn a dedicated session with the console output — do not patch speculatively.

## Item 2 — Stack page (`/stack`)

**Current state:** `src/routes/stack.tsx:1-10` renders a heading only.

**Design.** The v1 PRD called for a d3-force graph. That's a one-day build and it's the right long-term answer — but before it ships, the page should at minimum look intentional. Ship the static version first, replace with d3-force later without changing the route contract.

### Static v1 layout (ship first)

Five columns on desktop, stacking to one column on mobile, matching the canonical stack layering:

| Column | Technologies |
|---|---|
| **AI & Data Intelligence** | Hybrid RAG · Neo4j · pgvector · LangGraph · MCP · Vertex AI · Gemini · Claude Code |
| **Backend** | NestJS 11 · Node.js · Drizzle · Postgres · Redis/Bull · QStash · Cloud Pub/Sub |
| **Frontend** | Angular 21 · React 19 · Spartan UI · Tailwind v4 · TypeScript |
| **Infrastructure** | GCP · Terraform · Kubernetes · Docker · GitHub Actions · Firebase |
| **Compliance & Security** | HIPAA/BAA · AES-256-GCM · Google DLP · Tink · OAuth2/JWT |

Each technology is a card-sized pill with the tech name on the front. Clicking a pill expands an inline panel below the column with a **one-sentence operator note** describing *how Max uses this technology in production*. Not a Wikipedia entry. Examples:

- **Neo4j** → "Narrative causality graphs for memoir structure — nodes are story beats, edges are causal relationships, queried at inference time to keep AI suggestions internally consistent."
- **pgvector** → "Semantic recall layer for manuscript chunks — IVFFlat index at 1536 dimensions, queried alongside Neo4j for hybrid retrieval that beats either in isolation."
- **AES-256-GCM** → "Column-level PHI encryption via Drizzle middleware, keys rotated via GCP KMS with envelope encryption. Ciphertext never leaves the database; plaintext never leaves memory."

The operator-note copy is the moat. A bulleted list of buzzwords is what every other portfolio does; these one-liners prove Max works at the primitive level.

**Content source:** `src/content/stack.ts` — typed array of `{ group, name, note }`. Importable by both the static layout and the future d3-force graph so content doesn't need to migrate when the visualization upgrades.

**Interaction:** pill hover → subtle lift + accent border. Pill click → accordion-style inline expansion (not a modal, not a drawer — disclosure pattern keeps scroll context). Respect `prefers-reduced-motion`.

### Phase 2 upgrade (post-launch, not blocking)

Replace the grid with d3-force simulation per v1 PRD, using the same `stack.ts` content source. Out of scope for pre-cutover.

## Item 3 — Contact page (`/contact`)

**Current state:** `src/routes/contact.tsx:1-10` renders a heading only. Backend at `functions/src/contact.ts` exists and wires to Resend → `max.millien@puretome.com`.

**Scope:** frontend form that consumes the existing function. No backend changes.

### Form fields

- `name` — text, required, 2–80 chars
- `email` — email, required, RFC 5322 validated
- `company` — text, optional, 0–120 chars
- `message` — textarea, required, 20–4000 chars

All fields validated client-side with Zod (share the same schema as the function via a `shared/contact-schema.ts`). Form state via React Hook Form.

### Layout

Two-column desktop, stacked mobile:

- **Left column — the form** (fields as above, single submit button labeled `Send →`, subtle help text under each field)
- **Right column — direct channels:**
  - `max.millien@puretome.com` (displayed as text, not mailto, per PRD v1)
  - Calendly link (`Book a 30-min architecture call →`) reusing the CalendlyEmbed component *or* linking to `/engage#book` to avoid a second Calendly load
  - LinkedIn profile link
  - Location: Greater Boston

### Submission UX

- Inline field-level validation on blur
- Submit button disabled until the form is valid
- On submit: optimistic "Sending…" state (button spinner), then either:
  - Success → replace form with a "Message sent. I'll respond within 24 hours — `max.millien@puretome.com`" panel
  - Failure → red inline banner above the submit button with the error, form state preserved
- No captcha v1. Rate-limiting lives in the Firebase Function; if spam becomes a problem, add Cloudflare Turnstile later.

### Press inquiries

Per PRD v1, press inquiries route through `/author`, not here. Add a small footer note on `/contact`: *"Press and media inquiries: see the author page →"* linking to `/author#press`.

## Item 4 — OSS deep-dive content pass

**Current state:** `src/routes/work/oss.$slug.tsx:84-88` renders a placeholder: *"Threat model, design decisions, and the extraction note publish here in the next content pass. The README in the repo is the current source of truth."*

**Blocker for DNS cutover.** Per PRD-002 §"Change 5," the three OSS repos also need to exist publicly on GitHub before launch. Track both here.

### Per-repo deliverables

For each of `agent-native-monorepo`, `secure-data-vault`, `realtime-voice-infra`:

1. **Public GitHub repo under `github.com/maxmillienjr`** with:
   - `README.md` (architecture overview, stack, why-this-exists)

   - `CONTRIBUTING.md` or a "not accepting contributions yet" note
   - Working `package.json` / build (doesn't need to be feature-complete; does need to install and not throw)
   - A pinned issue or README section saying "v1.0 target: [date]"

2. **Site deep-dive at `/work/oss/[slug]`** covering:
   - **Problem** — the real-world need this pattern addresses (1–2 paragraphs)
   - **Architecture** — diagram (SVG, not a screenshot) + walkthrough
   - **Threat model / constraints** — what this protects against, what it explicitly doesn't
   - **Extraction note** — honest callout: "Extracted from [Quiet Horizons | PureTome] per the Sanitization Rule. Skeleton and conventions are real; domain logic is generic."
   - **Stack pills** reusing the Stack page component
   - **Open-source on GitHub →** CTA at the bottom

**Content source:** `src/content/oss/[slug].mdx` — one file per repo, imported by the dynamic route.

### Priority order

1. `secure-data-vault` first — it's the clearest narrative (compliance pain, clear before/after) and the smallest surface area to get right.
2. `realtime-voice-infra` second — pairs naturally with secure-data-vault (both extracted from Quiet Horizons).
3. `agent-native-monorepo` last — biggest scope, needs the most careful sanitization since it mirrors PureTome's real structure.
